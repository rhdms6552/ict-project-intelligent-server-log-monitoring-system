# Meeting Brief

## 프로젝트 개요

- 프로젝트명: Intelligent Server Log Monitoring System under Simulated Network Constraints
- 목표: 서버 로그 생성부터 네트워크 제약, 수집, 복구, 상태 판단, 대시보드 시각화까지 전체 파이프라인을 하나의 시스템으로 구현
- 개발 환경: Python 3.11, Streamlit, Windows

## 현재까지 진행한 내용

### 1. 데이터 생성

- `src/log_generator.py`에서 실시간 합성 서버 로그를 생성하도록 구현했습니다.
- 로그 형식은 `[YYYY-MM-DD HH:MM:SS] LEVEL message` 형식을 따릅니다.
- 로그 레벨 비율은 `INFO 60%`, `WARNING 25%`, `ERROR 12%`, `CRITICAL 3%`로 반영했습니다.
- `multiprocessing.Process`로 별도 프로세스에서 동작하게 구성했습니다.

### 2. 네트워크 제약 시뮬레이션

- `src/network_constraint.py`에서 소프트웨어 레벨의 제약을 구현했습니다.
- 지연, 드롭, 손상을 각각 독립적으로 반영합니다.
- 제약 파라미터는 `data/params.json`을 통해 실시간으로 읽도록 구성했습니다.
- 대시보드 슬라이더를 조작하면 메인 프로세스가 즉시 값을 반영합니다.

### 3. 로그 수집 및 처리

- `src/log_processor.py`에서 수신 로그를 버퍼링하고 통계를 집계합니다.
- 최근 20개 로그를 슬라이딩 윈도우로 유지합니다.
- 전체 로그는 최대 200개까지 유지합니다.
- `INFO`, `WARNING`, `ERROR`, `CRITICAL` 카운트와 수신/드롭/손상/복구 횟수를 관리합니다.
- 최근 60초 기준 에러율 시계열 데이터도 함께 저장합니다.

### 4. AI 기반 복구

- `src/ai_recovery.py`에서 `regex + difflib` 기반 복구 모듈을 구현했습니다.
- 손상된 로그 레벨 토큰을 정규식으로 추출하고, 비알파벳 문자 제거 후 알려진 레벨과 유사도 비교를 통해 복구합니다.
- 현재는 `INFO`, `WARNING`, `ERROR`, `CRITICAL` 레벨 복구를 지원합니다.

### 5. 최종 의사결정

- `src/decision_engine.py`에서 규칙 기반 상태 분류를 구현했습니다.
- 최근 20개 로그에 `CRITICAL`이 1개 이상이면 `CRITICAL` 상태입니다.
- `CRITICAL`이 없고 `ERROR`가 3개 이상이면 `WARNING` 상태입니다.
- 그 외에는 `STABLE` 상태입니다.
- 드롭률이 50% 이상이면 이상 점수에 패널티를 추가해 과도한 Stable 판정을 줄이도록 했습니다.

### 6. 라이브 대시보드

- `dashboard/dashboard.py`에서 Streamlit 대시보드를 구현했습니다.
- 현재 확인 가능한 항목:
- 시스템 상태 카드
- 이상 점수 progress bar
- 로그 레벨별 카운트
- 수신/드롭/손상/복구 통계
- 에러율 라인 차트
- 제약 슬라이더
- 최근 로그 스트림
- `data/state.json`을 읽어 1초 간격으로 새로고침되도록 구성했습니다.

## 현재 시점의 성과

- 문서 명세에 있는 6단계 필수 파이프라인은 모두 구현했습니다.
- 실제 실행도 완료했으며, 백엔드와 대시보드가 정상적으로 동작하는 것을 확인했습니다.
- `data/state.json`, `data/params.json` 자동 생성 및 갱신도 확인했습니다.
- 실시간 시뮬레이션과 시각화가 연결된 상태라 데모가 가능한 수준입니다.

## 미팅에서 솔직하게 말하면 좋은 부족한 점

### 1. 복구 기능의 범위가 아직 제한적입니다

- 현재 복구는 로그 전체 문장을 완벽하게 복원하는 방식이 아니라, 핵심인 로그 레벨 토큰 복구에 집중되어 있습니다.
- 즉, `ERROR`, `WARNING` 같은 상태 판단에 중요한 부분은 복구하지만, 메시지 본문까지 정교하게 복원하지는 못합니다.

### 2. 실제 네트워크 환경이 아닌 소프트웨어 시뮬레이션입니다

- 이번 프로젝트는 교육 목적에 맞춰 네트워크 제약을 코드 레벨에서 재현했습니다.
- 실제 패킷 레벨 제어나 분산 서버 환경에서의 실험은 아직 하지 않았습니다.

### 3. 의사결정 엔진이 규칙 기반입니다

- 현재는 해석 가능성과 구현 안정성을 위해 규칙 기반 분류를 사용했습니다.
- 추후에는 머신러닝 기반 이상 탐지나 패턴 분석으로 확장할 수 있습니다.

### 4. 자동 테스트와 예외 처리 보강이 더 필요합니다

- 현재는 수동 실행 검증은 했지만, 단위 테스트나 통합 테스트 코드는 아직 추가하지 않았습니다.
- 장시간 실행 시의 안정성, 예외 상황 복구, 파일 손상 대응은 더 강화할 필요가 있습니다.

### 5. 대시보드 저장성과 분석 기능은 확장 여지가 있습니다

- 지금은 실시간 관찰 중심입니다.
- 장기 이력 저장, 검색, 필터링, 알림 전송 기능은 아직 포함하지 않았습니다.

## 교수님께 설명할 때 쓸 수 있는 짧은 발표 스크립트

안녕하세요. 현재 프로젝트는 로그 생성, 네트워크 제약 시뮬레이션, 로그 수집, 손상 복구, 상태 판단, 대시보드 시각화까지 전체 파이프라인을 구현한 상태입니다.

실시간 로그는 별도 프로세스에서 생성되고, 중간에 지연, 드롭, 손상이 발생하도록 시뮬레이션했습니다. 손상된 로그는 `regex`와 `difflib`를 이용해 레벨 토큰을 복구하고, 최근 20개 로그를 기준으로 시스템 상태를 `STABLE`, `WARNING`, `CRITICAL`로 판단하게 했습니다.

현재 Streamlit 대시보드에서 상태, 이상 점수, 로그 카운트, 복구 통계, 에러율, 실시간 로그 스트림을 확인할 수 있고, 슬라이더로 제약 조건도 실시간 조절 가능합니다.

다만 아직 부족한 부분도 있습니다. 복구 기능이 메시지 전체 복원까지는 확장되지 않았고, 실제 네트워크 장비 수준이 아닌 소프트웨어 시뮬레이션이라는 한계가 있습니다. 또 현재 판단 엔진은 규칙 기반이라 향후에는 머신러닝 기반 이상 탐지로 확장할 계획입니다.

전체적으로는 데모 가능한 수준까지 구현을 완료했고, 앞으로는 복구 정확도, 테스트, 장기 이력 분석 기능을 보완하는 방향으로 발전시키려고 합니다.
